# spec-pipeline テンプレート

`planner` / `generator` / `evaluator` の3サブエージェントと `spec-pipeline` スキルによる自律開発フローを、新しい案件フォルダで即座に使えるようにするためのテンプレート。元ネタは `YouTubeアカウント自動運用` プロジェクトの構成。

## 中身

- `CLAUDE.md` — プロジェクト運用ルール(サブエージェントの役割分担・品質ゲート・リトライ上限など)。プロジェクト名等のハードコードなし、そのまま使える。
- `.claude/agents/planner.md` / `generator.md` / `evaluator.md` — 3サブエージェント定義
- `.claude/skills/spec-pipeline/SKILL.md` — オーケストレーションスキル本体
- `.claude/settings.json` — 権限設定
- `.mcp.json` — Playwright MCPサーバー定義(evaluatorの実機検証に必要)
- `.gitignore` — シークレット・依存関係・ビルド成果物などの除外設定
- `docs/dashboard.html` / `docs/pipeline-status.js` — 実行状況ダッシュボード。ブラウザで直接開くと、今どのエージェントが何をしているか・スプリント進捗・生成済みドキュメント(spec.md等)へのリンクが30秒間隔で自動更新される(サーバー不要)。「今すぐ更新」リンクで即時更新も可能。オーケストレーターが`docs/pipeline-status.js`を随時上書きする
- `docs/project-memory.md` / `docs/Home.md` / `docs/sprints/sprint-<N>-brief.md` — トークン節約とObsidian連携のための常設ファイル群。詳細は下記「トークン使用量を抑える運用」参照
- `docs/sprints/sprint-<N>-preview-*.png` — 画面を持つプロダクトでスプリントがPASSした際にevaluatorが保存する進捗プレビュー画像(任意)。詳細は下記「全自動化のための機能」参照

## 新しい案件で使う手順

1. 新しい案件用のフォルダを作る(例: `C:\ClaudeProjects\<新案件名>\`)
2. このテンプレートフォルダの中身一式(隠しファイル含む)をコピーする

   ```powershell
   Copy-Item -Path "C:\ClaudeProjects\_templates\spec-pipeline\*" -Destination "C:\ClaudeProjects\<新案件名>\" -Recurse -Force
   Copy-Item -Path "C:\ClaudeProjects\_templates\spec-pipeline\.gitignore" -Destination "C:\ClaudeProjects\<新案件名>\.gitignore" -Force
   Copy-Item -Path "C:\ClaudeProjects\_templates\spec-pipeline\.claude" -Destination "C:\ClaudeProjects\<新案件名>\.claude" -Recurse -Force
   Copy-Item -Path "C:\ClaudeProjects\_templates\spec-pipeline\.mcp.json" -Destination "C:\ClaudeProjects\<新案件名>\.mcp.json" -Force
   ```

   (`docs/dashboard.html`・`docs/pipeline-status.js`は`*`のワイルドカードコピーに含まれる通常ファイルなので個別コピーは不要)

   (`Copy-Item -Recurse` はワイルドカード指定時に隠しファイル・隠しフォルダ(`.claude`・`.gitignore`・`.mcp.json`)を取りこぼす場合があるため、上記のように個別にもコピーするのが確実)

3. コピー先フォルダで `git init`(gitリポジトリ化。spec-pipelineはgit差分ベースで動くため必須)
4. `git config user.name` / `user.email` が未設定ならユーザー自身が設定
5. 新しいセッションでそのフォルダを開き、`/spec-pipeline <1〜4行のアイデア>` で開始

## トークン使用量を抑える運用（Obsidian / NotebookLM）

このパイプラインは複数の仕組みでClaude Codeのトークン消費を抑える。**いずれも要約による情報欠落や検証の手抜きではないため、成果物のクオリティ（実装・検証の精度）は落ちない**。曖昧さを減らす仕組み（受け入れ基準の具体化等）はむしろ誤判定・手戻りを減らし品質にも寄与する。

**自動・エンジン内蔵（設定不要、常に有効）**

1. **仕様書の原文抜粋（sprint-brief）**: `generator`/`evaluator`は毎スプリント仕様書全体を読み直す代わりに、対象スプリント分だけを原文のまま抜粋した`docs/sprints/sprint-<N>-brief.md`を読む。オーケストレーターが仕様書を1回読むだけで全スプリント分をまとめて生成するため、サブエージェント側の入力トークンが大きく減る。
2. **不合格理由の要点転記**: リトライ時、evaluatorレポートや`/security-review`の指摘を全文渡すのではなく、オーケストレーターが「発見したバグ・問題点」やCritical/High指摘そのものだけを次のgenerator呼び出しに転記する。
3. **技術選定メモの上書き運用**: `docs/spec/<slug>-architecture.md`は同じ領域を再検討した際に追記でなく上書きするため、長期プロジェクトでも際限なく肥大化しない。
4. **簡潔なレポート形式・具体的な受け入れ基準**: 各種レポートは長いプローズでなく箇条書き。受け入れ基準は曖昧さのない検証可能な文で書かれ、無駄な再実装サイクルを減らす。

**任意（設定・利用はユーザー次第）**

5. **Obsidian（無料、ローカル完結）**: プロジェクトフォルダ（または`docs/`）を[Obsidian](https://obsidian.md/)のVaultとして開くと、`docs/Home.md`を入口に仕様書・技術選定メモ・各スプリントの自己評価/評価レポート・`docs/project-memory.md`がwikilink/backlinkで繋がって閲覧できる。「スプリント3で何をやったか」「なぜこの技術を選んだか」等をClaude Codeのチャットで聞き直す代わりにObsidianで無料に確認できる。追加設定は不要（生成済みMarkdownがそのままVaultになる）。
6. **NotebookLM（手動）**: [NotebookLM](https://notebooklm.google.com/)には公開APIが無いため、Claude Codeからの自動連携はできない。トークンを使わずに自然言語でのQ&A・要約・音声概要を作りたい場合は、`docs/project-memory.md`（と必要に応じて`spec.md`）をユーザー自身が手動でNotebookLMにアップロードする。得られた結論（技術的な決定事項など）をClaude Codeに反映したい場合は、そのファイルに追記してからスプリントを再開する。

## 品質を高める無料機能（UI可視化・企画・実装レビュー）

以下は「無料または追加コストなしで使える範囲で、品質向上に寄与するもの」を検討した上で採用したもの。検討したが不採用にしたものも参考までに記載する。

**採用**

- **画面ワイヤーフレーム（自動・設定不要）**: 対象プラットフォームが画面を持つ場合、`planner`が主要画面のグレースケール・ワイヤーフレーム（配色・実装は含まない、箱とラベルのみ）を`docs/spec/<slug>-wireframes.html`として仕様書と一緒に生成する。generatorが実装する前に画面イメージを共有でき、認識齟齬による手戻りを減らす。
- **ブレインストーミング（自動・アイデアが曖昧な場合のみ）**: 新規アイデアが短く、対象プラットフォーム・ターゲットユーザー・スコープの境界などが読み取れない場合、`planner`を呼ぶ前にオーケストレーターがAskUserQuestionで2〜3問だけ確認する。詳細なアイデアには質問しない。
- **`/code-review`（自動・Claude Code標準機能、追加コストなし）**: `evaluator`のPASS後、`/security-review`と並行して`/code-review`（effort: low）を同じ差分に実行する。機能テストにもセキュリティレビューにも含まれない「正しさのバグ」「簡潔化・効率化の余地」を検出する追加ゲート。

**検討したが不採用**

- **Claude Design**（Anthropic Labs、2026年4月〜）: デザイン・プロトタイプ作成ツールだが、Pro/Max/Team/Enterprise等の有料プラン限定で無料枠がなく、API/MCP等の外部呼び出し口も見当たらないため、無人パイプラインへの自動組み込みは不可能と判断。
- **Pencil**（[pencil.evolus.vn](https://pencil.evolus.vn/)）: 無料・オープンソースのワイヤーフレームツールだが、保存形式(.epgz)が独自のXML+gzipでスキーマが薄く公開されており、コード生成すると壊れたファイルを作るリスクがあるため、素のHTMLワイヤーフレームの方が安全で同じ目的を達成できると判断。
- **Superpowers**（[obra/superpowers](https://github.com/obra/superpowers)）: Claude Code向けの別の開発方法論一式（ブレインストーミング→計画→サブエージェント駆動実装→検証）。コンセプトはこのテンプレートと重なるが、丸ごと導入するとplanner/generator/evaluatorの役割分担設計と衝突しうるため不採用。ただし「曖昧なアイデアに質問する」「タスクを小さく分解する」という考え方は、このテンプレート自身の設計として上記に反映済み。

## 全自動化のための機能（環境準備・事前確認・進捗プレビュー）

このパイプラインをできるだけ人手を介さず最後まで走らせるための仕組み。

- **環境不足の自動補完（自動・常に有効）**: ビルド・起動・テストが「無料でプロジェクトローカルに閉じるツール・パッケージが足りない」ことだけを理由に失敗する場合（例: Playwrightのブラウザ本体が未インストール）、`generator`/`evaluator`は「未検証」として諦める前に自動でインストールしてから再試行する。これにより実機検証できず「未検証」扱いになる項目を減らす。**OS全体に影響するインストール（管理者権限が必要なもの・グローバルなインストーラー実行等）は対象外**とし、非対話環境で権限昇格待ちのまま止まることを防ぐ。該当する場合は自己評価/評価レポートに手動インストール手順を明記するに留める。
- **外部サービス依存・管理者権限インストール予想の事前確認（自動・該当する場合のみ）**: スプリントループを開始する前に、オーケストレーターが次の2点を確認しAskUserQuestionで一度（両方あれば一度にまとめて）だけ質問する。個々のスプリントの途中で初めて発覚して手が止まる、という事態を避けるため、判明・予想できるものは最初にまとめて聞く設計。
  - 仕様書に外部API連携・OAuth認証・決済等、ユーザー自身の操作（ログイン・APIキー発行・アカウント登録等）が必要な項目が含まれる場合:「今すぐ認証情報等を用意する」か「揃うまでモック実装で進め、後で本接続に切り替える」かを選べる。
  - 対象プラットフォームが`デスクトップ`、または機能一覧に動画/音声処理・重量級ローカルDB・OS常駐等が含まれ、環境不足の自動補完では対応できない**管理者権限インストールが予想される**場合:「事前にインストールしておく」「管理者権限が不要な代替手段（クラウド/Docker/純npm実装等）で進める」「実際にその場面が来たら都度判断する」から選べる。これはあくまで予想であり、外れても実害はない（実際に発生した時点の通常フォールバックに委ねるだけ）。
- **進捗プレビュー画像（自動・画面を持つプロダクトのみ）**: 各スプリントがPASSするたびに、`evaluator`が直前まで検証していた画面の状態から追加のブラウザナビゲーションなしでスクリーンショットを1〜3枚`docs/sprints/sprint-<N>-preview-*.png`として保存し、ダッシュボードのスプリントカードにサムネイル表示する。保存に失敗してもスプリントの合否・進行には一切影響しない。動画や対話的な試作品の自動生成は行わないが、各スプリントはPASSした時点でコミットされた動くアプリそのものであり、README記載の起動コマンドでいつでも本人が直接触って確認できる。
- **一時的な基盤障害への耐性（自動・常に有効）**: 「temporarily unavailable」等、ツール呼び出し基盤側の一時的な瞬断はコードの不具合と区別し、スプリントのリトライ上限を消費させずに同じ呼び出しを自動で再試行する（このテンプレート自体の開発中にも実際に発生し、再試行だけで解消した実績がある）。基盤障害を理由にスプリントをFAILや要人手介入にはしない。

## 注意事項

- このテンプレートを更新した場合、元プロジェクト側の `.claude/agents/*.md` や `SKILL.md` を手動で編集しても自動反映はされない。改善を横展開したい場合は都度このテンプレートにもコピーし直す。
- 1つのリポジトリに複数の無関係な案件を混在させると `git add -A` ベースのスプリントコミットが汚染されるため、**案件ごとに独立したフォルダ(独立したgitリポジトリ)にする**のが前提。
