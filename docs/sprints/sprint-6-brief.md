# Sprint 6 brief: X へ実際にスレッド投稿できる

## 対象プラットフォーム
`web`(画面を持たないバックエンド自動化)。

## 含まれる機能(仕様書「機能一覧」より原文)

### F6. X への投稿実行(スレッド連結)
- 説明: 生成・分割された一連のツイートを、Xに実際に投稿する。2件目以降は直前のツイートへの返信として繋ぎ、1本のスレッドにする。
- 優先度: Must
- 受け入れ基準:
  - [ ] 認証情報が正しく設定されていれば、単一ツイートおよびスレッド(複数ツイート)を投稿できる。
  - [ ] スレッドの2件目以降が直前ツイートへの返信として連結され、Xのタイムライン上で1本のスレッドとして表示される。
  - [ ] 途中のツイート投稿が失敗した場合、失敗箇所を記録し、部分的に投稿された状態を把握できる(どこまで投稿されたかがログで分かる)。
  - [ ] 投稿成功時、投稿したツイートID群と投稿時刻が記録される。
  - [ ] X APIのレート制限・上限エラーを検知した場合は、リトライを規約の範囲に留め、超過時は諦めて理由を記録する(規約違反の連投をしない)。

## このスプリントの受け入れ基準(デモできる状態・仕様書「スプリント計画」より原文)
- [ ] 正しい認証情報下で、単一ツイートおよび複数ツイートのスレッド投稿ができる(認証情報が用意できない検証環境では、投稿API呼び出しをモック/スタブ化し、正しい順序・返信連結の呼び出しが行われることを確認する。検証モードをレポートに明記)。
- [ ] スレッド2件目以降が直前ツイートへの返信として連結される呼び出しになっている。
- [ ] 途中失敗時にどこまで投稿されたかが記録される。
- [ ] レート制限/上限エラー検知時に、規約範囲のリトライに留め、超過時は諦めて記録する。
- [ ] 投稿成功時にツイートID群と時刻が記録される。

## 評価基準(仕様書より原文)
- 致命的バグ: 0件 / コンソールエラー: 0件 / 受け入れ基準充足率: 100%
- 補足: 本スプリントは実アカウントへの投稿がレート制限・規約リスクを伴うため、既定はモック検証+任意で1回の実投稿確認とし、無闇な連投検証はしない。

## 関連するリスク・留意事項(仕様書より原文抜粋・外部サービス依存)
- **X API の認証情報発行**: 投稿には X の API Key/Secret、Access Token/Secret(および必要に応じてアプリ権限設定)が必要。これらは利用者本人が X Developer Portal でアプリを作成し発行する必要があり、システムが代行できない。Access Token には「書き込み(Read and Write)」権限が必要。
- **X API 利用規約・レート制限・投稿数上限**: X API の無料枠(Free)/Basic 等のティアによって、1アカウントあたり・時間あたり・月あたりの投稿(create)回数に上限がある。ティアによっては1日3回×スレッド複数ツイートという投稿量が無料枠の月間上限に抵触しうるため、利用者は自分の契約ティアの上限を確認し、必要なら上位ティアを検討する必要がある。上限超過時はシステム側でエラーとして扱い、投稿を諦めて記録する(規約違反となる連投・回避策は行わない)。自動投稿・スパム的挙動は X の自動化ポリシーに抵触しうるため、投稿頻度・内容は規約の範囲に収める。
- 認証情報は`.env`等の設定値経由でのみ読み込み、リポジトリには実値を残さない。

## 事前確認の結果(オーケストレーターがユーザーに確認済み)
X API認証情報(API Key/Secret, Access Token/Secret)は、過去の類似プロジェクト(x-auto-post)のGitHub Actions Secretsに登録済みのものを再利用する予定。配置先リポジトリは既存の `github.com/menx8374-source/x-auto-post` を新システムで置き換える方針(ユーザー確定済み)。ローカルの`.env`にまだ値が無い検証環境では、上記の受け入れ基準通りモック/スタブ検証で進めてよい。
