# Sprint 7 brief: 投稿状態管理と冪等性・不発リカバリができる

## 対象プラットフォーム
`web`(画面を持たないバックエンド自動化)。

## 含まれる機能(仕様書「機能一覧」より原文)

### F9. 投稿状態・履歴管理と冪等性(不発リカバリ)
- 説明: 何をいつ投稿したかを永続的に記録し、同じ枠の二重投稿を防ぎつつ、トリガー不発時に取りこぼしを補えるようにする。
- 優先度: Must
- 受け入れ基準:
  - [ ] 投稿した記事URL・投稿枠・投稿日時・ツイートIDが履歴として永続化され、次回実行時に参照できる。
  - [ ] 同一枠・同一日に対して処理が二重に起動しても、二重投稿されない(1枠1投稿の冪等性が保証される)。
  - [ ] ある枠のトリガーが不発だった場合でも、次の起動時に「当日その枠が未投稿」と判定でき、規定の許容範囲内なら投稿を補える(遅れての投稿可否は設定で制御でき、深夜に朝枠を投稿するような無制限な遅延投稿はしない)。
  - [ ] 履歴には過去分が一定期間ぶん残り、F2の既出判定に利用される。

## このスプリントの受け入れ基準(デモできる状態・仕様書「スプリント計画」より原文)
- [ ] 投稿履歴(記事URL・枠・日時・ツイートID)が永続化され、次回実行で参照される。
- [ ] 同一枠・同一日で処理を二重起動しても二重投稿されない(擬似二重起動で確認)。
- [ ] ある枠が未投稿のまま次の起動が来たとき「当日その枠が未投稿」と判定でき、許容範囲内なら補い、範囲外(例: 深夜に朝枠)なら補わない。
- [ ] 履歴が一定期間残り、既出判定に使われる。

## 評価基準(仕様書より原文)
- 致命的バグ: 0件 / コンソールエラー: 0件 / 受け入れ基準充足率: 100%

## 補足
Sprint 2で実装した「既出判定に必要な最小の履歴参照」を、このスプリントで正式なF9(冪等性・不発リカバリを含む完全な履歴管理)に拡張する。Sprint 2の実装と矛盾しないよう既存の履歴データ形式との後方互換に配慮する。

## 関連するリスク・留意事項(仕様書より原文抜粋)
- GitHub Actions の schedule(cron)トリガーは高負荷時に遅延・不発することが知られているため、定時起動の主トリガーには使わず、外部cronサービスからの workflow_dispatch 呼び出しを主経路とする。ただし外部cronサービス自体も無料枠の可用性保証はないため、不発時にリカバリできるよう「その時間帯に未投稿なら次の起動で補う」等の冪等・回復設計を仕様に含める。
