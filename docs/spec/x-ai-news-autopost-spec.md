# X AIニュース自動投稿システム 製品仕様書

## 概要
X(旧Twitter)上で、Web上で今まさに注目度が急上昇している最新のAIニュースを自動で選び、要約したポストを1日3回の高インプレッション時間帯に自動投稿するバックエンド自動化システム。文字数制限を厳守し、収まらない場合はスレッド分割し、元記事リンクを別ツイートとして付与する。運用者は自分のXアカウントを運用する個人1名で、投稿の自動生成・自動投稿だけを目的とする(画面・複数ユーザー機能・収益化機能は持たない)。

## 対象プラットフォーム
`web`(画面を持たないバックエンド自動化)。ユーザーが明示している通り、UIスクリーンを持たず、外部トリガーによって定時起動され投稿処理を行うサーバーサイド/バッチ型の自動化システムとして構築する。ダッシュボードやフォーム等の操作画面は本バージョンでは提供しない。

## ターゲットユーザー
- 自分1名でXアカウントを運用する個人。AIニュースを定期的に発信したいが、毎日手動で話題を探して投稿する時間をかけたくない。
- 技術リテラシーはある程度あり、GitHubアカウント・各種APIキーの発行・外部cronサービスの初期設定を自分で行える。
- 複数アカウント運用・チーム運用・他ユーザーへの提供は想定しない(シングルユーザー・シングルアカウント前提)。

## スコープ外(Non-Goals)
- 操作用のWeb画面・管理ダッシュボード・ログイン機能(本システムは無人バッチのため画面を持たない)。
- 複数Xアカウント・複数ユーザーへの提供、マルチテナント。
- 投稿へのリプライ対応・エンゲージメント返信・DM対応などの双方向コミュニケーション。
- 画像・動画・GIFの生成や添付(初期スコープはテキスト投稿とリンクのみ)。
- 過去の投稿実績データに基づく投稿時刻の自動最適化(初期スコープは固定の目安時刻3回。将来拡張余地としては残すが本バージョンでは実装しない)。
- 収益化・広告・アフィリエイトリンクの挿入。
- AIニュース以外のジャンル(一般ニュース・雑多なノウハウ/意見ツイート)の投稿。
- 投稿後のパフォーマンス分析レポートの自動生成。

## リスク・留意事項
本システムは複数の外部サービス連携と認証情報の取り扱いを伴い、いくつかは利用者自身の操作でしか用意できない。以下は"何を作るか"の一部として無視できない制約である。

### 利用者自身の操作が必要な外部サービス設定
- **X API の認証情報発行**: 投稿には X の API Key/Secret、Access Token/Secret(および必要に応じてアプリ権限設定)が必要。これらは利用者本人が X Developer Portal でアプリを作成し発行する必要があり、システムが代行できない。Access Token には「書き込み(Read and Write)」権限が必要。
- **X API 利用規約・レート制限・投稿数上限**: X API の無料枠(Free)/Basic 等のティアによって、1アカウントあたり・時間あたり・月あたりの投稿(create)回数に上限がある。ティアによっては1日3回×スレッド複数ツイートという投稿量が無料枠の月間上限に抵触しうるため、利用者は自分の契約ティアの上限を確認し、必要なら上位ティアを検討する必要がある。上限超過時はシステム側でエラーとして扱い、投稿を諦めて記録する(規約違反となる連投・回避策は行わない)。自動投稿・スパム的挙動は X の自動化ポリシーに抵触しうるため、投稿頻度・内容は規約の範囲に収める。
- **GitHub Personal Access Token(PAT)の発行**: 外部cronサービスから GitHub Actions の workflow_dispatch を叩くために、workflow 起動権限を持つ PAT を利用者本人が発行する必要がある。PAT は有効期限が切れると外部トリガーが不発になるため、期限管理は利用者の責任範囲。
- **cron-job.org(または同等の無料外部cronサービス)の設定**: 定時トリガーの実体として、利用者本人がアカウント登録し、3つの時刻それぞれに対して GitHub の workflow_dispatch エンドポイントを叩くジョブ(URL・HTTPメソッド・認証ヘッダ・リクエストボディ)を設定する必要がある。設定手順は本システムがドキュメントとして提供するが、登録・設定操作自体は利用者が行う。
- **Anthropic API(Claude)のAPIキー発行**: ニュース要約・文面生成に Claude を用いる場合、利用者本人が Anthropic のアカウントで API キーを発行し、従量課金(有料)を負担する必要がある。キー未設定では文面生成が行えない。

### 認証情報の取り扱い
- 上記すべての認証情報(X の各キー、GitHub PAT、Anthropic API キー)は機密情報であり、ソースコードや公開リポジトリに平文で含めてはならない。設定値経由でのみ読み込み、リポジトリには実値を残さず、キー名のみの雛形を残す運用とする(具体的な保管方法は実装側の裁量)。
- GitHub Actions で実行する場合、これらのシークレットは GitHub の Secrets 機能等に利用者本人が登録する必要がある。

### スケジュール実行の信頼性
- GitHub Actions の schedule(cron)トリガーは高負荷時に遅延・不発することが知られているため、定時起動の主トリガーには使わず、外部cronサービスからの workflow_dispatch 呼び出しを主経路とする。ただし外部cronサービス自体も無料枠の可用性保証はないため、不発時にリカバリできるよう「その時間帯に未投稿なら次の起動で補う」等の冪等・回復設計を仕様に含める(F9参照)。

## 機能一覧

### F1. 急上昇AIニュースの収集とスコアリング
- 説明: Web上のAIニュース情報源から直近の記事候補を収集し、「今まさに注目度が急上昇している」度合いをスコア化して候補リストを作る。単なる新着ではなく、直近で話題性が急伸しているものを上位に並べる。
- 優先度: Must
- 受け入れ基準:
  - [ ] 実行すると、少なくとも10件以上のAIニュース候補が「タイトル・元記事URL・情報源・推定公開時刻・急上昇スコア」を持つ構造化データとして得られる。
  - [ ] 各候補に急上昇スコアが付与され、候補リストがスコア降順に並ぶ。
  - [ ] スコアリングは「新しさ(直近性)」と「話題の伸び(複数ソースでの言及・急な言及増加等の話題性シグナル)」の両方を反映し、公開が古く話題も伸びていない記事は上位に来ない(検証: 意図的に古い低話題の記事を混ぜても上位3件に入らない)。
  - [ ] 収集対象がAI関連(生成AI・LLM・機械学習・AI製品/企業/研究等)に限定され、無関係ジャンルの記事は候補に含まれない。
  - [ ] 情報源への通信失敗時も処理全体は落ちず、取得できたソース分だけで候補リストを構築し、失敗したソースはログに記録される。

### F2. 投稿ニュースの選定(重複・既出回避)
- 説明: F1の候補リストから、その回に投稿する1件を選定する。過去に投稿済みの記事や、同一トピックの重複を避けて選ぶ。
- 優先度: Must
- 受け入れ基準:
  - [ ] 実行すると、候補リストから投稿対象の1件が確定し、その記事のタイトル・URL・選定理由(スコア)がログに残る。
  - [ ] 過去に投稿済みの記事URL(および実質同一記事)は選定対象から除外される(検証: 直前に投稿済みとして記録したURLは次回選定で選ばれない)。
  - [ ] 同一日の3回の投稿で、可能な限り異なるトピックが選ばれる(同じ記事が同日中に2回選ばれない)。
  - [ ] 有効な候補が1件もない場合(全件が既出/しきい値未満)は投稿をスキップし、その旨を理由付きでログに記録する(空投稿・低品質投稿をしない)。

### F3. ポスト文面の生成
- 説明: 選定した記事をもとに、日本語で読みやすく、そのニュースの要点が伝わる投稿文面を生成する。煽りすぎず正確で、AIニュースアカウントとして自然なトーン。
- 優先度: Must
- 受け入れ基準:
  - [ ] 実行すると、選定記事に対応した日本語の投稿本文が生成される。
  - [ ] 本文は記事の要点(何が・どう新しいか)を含み、記事タイトルの丸写しではなく要約・言い換えになっている。
  - [ ] 事実に基づき、記事に書かれていない断定的な誇張や虚偽情報を含まない(検証: 生成文の主張が元記事の内容の範囲に収まっている)。
  - [ ] トーン・言語(日本語)は設定で一貫しており、毎回極端にブレない。
  - [ ] 文面生成に失敗した場合(APIエラー等)は投稿せず、エラーを記録して安全に終了する(壊れた/空の投稿をしない)。

### F4. 文字数チェックとスレッド自動分割
- 説明: X非Premiumの文字数上限(半角1・全角2換算で280相当、日本語は概ね140文字目安)を厳守し、1ツイートに収まらない場合は複数ツイートのスレッドに自動分割する。
- 優先度: Must
- 受け入れ基準:
  - [ ] 生成文面の文字数を、半角1・全角2換算のXの計算方式に合わせて算出し、上限内かどうか判定できる。
  - [ ] 上限内なら単一ツイート、超過するなら複数ツイートに分割され、各ツイートが個別に上限を超えない。
  - [ ] 分割は文の途中(単語や文の不自然な位置)で切れず、意味のまとまりを保った区切りで行われる。
  - [ ] スレッドの各ツイートに順序が分かる表記(例: 1/3, 2/3 等)が付与され、かつその表記分を含めて各ツイートが上限内に収まる。
  - [ ] 極端に長い元テキストでも無限にツイートが増えないよう上限本数の目安があり、超える場合は要約を短くする等で規定本数内に収める。

### F5. 元記事リンクの付与
- 説明: 元のAIニュース記事へのリンクを、本文とは別のツイート(スレッドの末尾等)として付与できる。
- 優先度: Must
- 受け入れ基準:
  - [ ] スレッドの末尾(または規定位置)に、元記事URLを含むリンク用ツイートが1つ追加される。
  - [ ] 本文ツイート側は文字数計算にURLを含めずに済み(リンクが別ツイートのため)、本文が上限内に収まる。
  - [ ] URLを含むツイートも、Xのリンク文字数換算方式を考慮して上限内に収まる。
  - [ ] 単一ツイートで本文が収まる場合でも、リンクツイートは別ツイートとして続けて投稿され、リプライチェーン(スレッド)として繋がる。

### F6. X への投稿実行(スレッド連結)
- 説明: 生成・分割された一連のツイートを、Xに実際に投稿する。2件目以降は直前のツイートへの返信として繋ぎ、1本のスレッドにする。
- 優先度: Must
- 受け入れ基準:
  - [ ] 認証情報が正しく設定されていれば、単一ツイートおよびスレッド(複数ツイート)を投稿できる。
  - [ ] スレッドの2件目以降が直前ツイートへの返信として連結され、Xのタイムライン上で1本のスレッドとして表示される。
  - [ ] 途中のツイート投稿が失敗した場合、失敗箇所を記録し、部分的に投稿された状態を把握できる(どこまで投稿されたかがログで分かる)。
  - [ ] 投稿成功時、投稿したツイートID群と投稿時刻が記録される。
  - [ ] X APIのレート制限・上限エラーを検知した場合は、リトライを規約の範囲に留め、超過時は諦めて理由を記録する(規約違反の連投をしない)。

### F7. 1日3回・固定時刻でのスケジュール投稿
- 説明: 高インプレッションが見込める時間帯として、1日3回の固定目安時刻に投稿処理を走らせる。時刻はXのゴールデンタイムのビジネスTZ(日本の一般的傾向)に基づき本仕様で定める。
- 優先度: Must
- 受け入れ基準:
  - [ ] 投稿時刻は日本時間(JST)基準で朝・昼・夜の3枠とし、初期値を **07:30 / 12:15 / 21:00 JST** とする(通勤/始業前・昼休み・夜のピーク帯を狙う目安)。
  - [ ] 各枠は「その枠向けの1投稿」を行い、1日で最大3投稿になる(同枠内で二重に投稿しない)。
  - [ ] どの枠として実行されたか(朝/昼/夜)が判別・記録され、枠ごとに独立して当日の投稿済み状態を持つ。
  - [ ] 目安時刻は将来変更しやすいよう1箇所の設定で管理され、コード各所に散らばっていない(F12と連動)。

### F8. 外部cronサービスからのトリガー連携
- 説明: GitHub Actions の schedule(cron)は遅延・不発が多いため、cron-job.org 等の外部無料サービスから GitHub Actions の workflow_dispatch エンドポイント(REST API)を定時に叩いて起動する。この方式のための起動口と設定手順を提供する。
- 優先度: Must
- 受け入れ基準:
  - [ ] 外部から workflow_dispatch で起動できる実行ワークフローが用意され、手動での workflow_dispatch 発火で投稿処理(またはドライラン)が走ることを確認できる。
  - [ ] ワークフローは「どの投稿枠(朝/昼/夜)か」を入力として受け取れ、外部cron側で枠ごとに別ジョブとして呼び分けられる(または起動時刻から枠を判定できる)。
  - [ ] cron-job.org 等で3枠ぶんのジョブを設定するための手順書(叩くURL・HTTPメソッド・必要な認証ヘッダ・リクエストボディの形・設定すべき時刻)がドキュメントとして提供される。
  - [ ] 認証情報(GitHub PAT等)を手順書に平文で書かず、利用者が自分の値を差し込む前提のプレースホルダで示す。
  - [ ] GitHub Actions の schedule トリガーは主経路にせず、あってもバックアップ用途である旨がドキュメントに明記される。

### F9. 投稿状態・履歴管理と冪等性(不発リカバリ)
- 説明: 何をいつ投稿したかを永続的に記録し、同じ枠の二重投稿を防ぎつつ、トリガー不発時に取りこぼしを補えるようにする。
- 優先度: Must
- 受け入れ基準:
  - [ ] 投稿した記事URL・投稿枠・投稿日時・ツイートIDが履歴として永続化され、次回実行時に参照できる。
  - [ ] 同一枠・同一日に対して処理が二重に起動しても、二重投稿されない(1枠1投稿の冪等性が保証される)。
  - [ ] ある枠のトリガーが不発だった場合でも、次の起動時に「当日その枠が未投稿」と判定でき、規定の許容範囲内なら投稿を補える(遅れての投稿可否は設定で制御でき、深夜に朝枠を投稿するような無制限な遅延投稿はしない)。
  - [ ] 履歴には過去分が一定期間ぶん残り、F2の既出判定に利用される。

### F10. 実行ログとエラー通知
- 説明: 各実行の結果(選定した記事・投稿本数・成功/失敗・スキップ理由)を記録し、失敗時に利用者が気づけるようにする。
- 優先度: Should
- 受け入れ基準:
  - [ ] 各実行で「開始・候補件数・選定記事・生成本文・投稿結果(成功/失敗/スキップ)・所要のエラー内容」がログとして残る。
  - [ ] 投稿失敗や連続スキップなど、利用者の対応が必要な事象が発生した場合に通知(実行基盤のログ/メール/任意のWebhookのいずれかでよいがドキュメントでどれかを案内)される仕組みがある。
  - [ ] ログに認証情報の実値が出力されない(トークン等がマスクまたは非出力)。

### F11. ドライラン(投稿せずプレビュー)
- 説明: 実際にXへ投稿せずに、「その回に何を投稿する予定か」(選定記事・生成本文・分割後の各ツイート・リンクツイート)を確認できるモード。テスト・レート制限節約・規約リスク回避に使う。
- 優先度: Must
- 受け入れ基準:
  - [ ] ドライランを指定して実行すると、Xへは1件も投稿されず、投稿予定の全ツイート(順序・各文字数・リンクツイート含む)が出力される。
  - [ ] ドライランでも記事収集・選定・文面生成・文字数計算・分割まで本番と同じ処理を通り、本番投稿時との差異が「実際に送信するか否か」だけである。
  - [ ] ドライランでは投稿履歴(既出判定用)を汚さない、または汚さないよう選べる。

### F12. 設定管理
- 説明: 投稿時刻・言語/トーン・1回の最大ツイート本数・スレッド分割の閾値・情報源リスト・許容遅延など、運用パラメータを一元的に設定できる。
- 優先度: Should
- 受け入れ基準:
  - [ ] 投稿枠の時刻、言語/トーン、最大ツイート本数、リンクツイートの有無/位置、許容遅延時間などが1箇所の設定で変更でき、変更が処理に反映される。
  - [ ] 認証情報系の設定と、挙動系の設定が分離され、認証情報はリポジトリに残さない前提の管理になっている。
  - [ ] 設定に不正値(未設定の必須項目・不正な時刻等)がある場合、起動時に検知して分かるエラーを出し、壊れた挙動のまま投稿しない。

## スプリント計画

### Sprint 1: 急上昇AIニュースを収集・スコアリングできる
- 含まれる機能: F1
- 受け入れ基準(このスプリント終了時にデモできる状態):
  - [ ] コマンド1つで実行すると、10件以上のAIニュース候補が「タイトル・URL・情報源・推定公開時刻・急上昇スコア」を持ってスコア降順で出力される。
  - [ ] 意図的に「古く話題も伸びていない記事」を混ぜても上位3件に入らないことを確認できる。
  - [ ] AI無関係の記事が候補に混ざらない。
  - [ ] 一部の情報源が通信失敗しても処理は落ちず、取得できた分で候補を作りエラーはログに残る。
- 評価基準(エバリュエーターが確認すべき閾値):
  - 致命的バグ: 0件
  - コンソールエラー: 0件(想定内のソース通信失敗をハンドリングしたログ出力は「エラー」に数えない)
  - 上記受け入れ基準の充足率: 100%

### Sprint 2: 投稿対象1件を既出回避して選定できる
- 含まれる機能: F2(+ F9の既出判定に必要な最小の履歴参照)
- 受け入れ基準:
  - [ ] Sprint1の候補リストから投稿対象1件が確定し、タイトル・URL・選定スコアがログに出る。
  - [ ] 直前に投稿済みとして記録したURLは次回選定で選ばれない。
  - [ ] 有効候補が0件のとき、投稿対象なしとして理由付きでスキップされる。

## 用語集
- **投稿枠(朝/昼/夜)**: 1日3回の固定投稿タイミング。初期値 JST 07:30 / 12:15 / 21:00。
- **急上昇スコア**: 記事の「直近性」と「話題の伸び(言及の急増等)」を反映した、その記事が今注目されている度合いの指標。値が高いほど優先して投稿対象になる。
- **スレッド**: 連続した複数ツイートを返信で繋いだ1本の投稿。文字数に収まらない本文を分割する際に用いる。
- **リンクツイート**: 元記事URLを含む、本文とは別の単独ツイート。スレッド末尾等に付与する。
- **ドライラン**: 実際にXへ投稿せず、投稿予定内容だけを出力する検証モード。
- **workflow_dispatch**: GitHub Actions のワークフローを外部から手動/API起動するためのトリガー。本システムの主起動経路。
