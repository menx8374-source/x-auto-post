# プロジェクト概要

このプロジェクトのアプリ開発は、`planner` / `generator` / `evaluator` の3サブエージェント(`.claude/agents/`)を `spec-pipeline` スキル(`.claude/skills/spec-pipeline/SKILL.md`)がオーケストレーションする自律開発フローで進める。

## 起動方法

- 新規アイデアから始める場合: `/spec-pipeline <1〜4行のアイデア>`
- 既存specの途中スプリントから再開する場合: `/spec-pipeline <docs/spec/配下のspec.mdへのパス> [開始スプリント番号]`

## 3サブエージェントの役割分担(越境禁止)

| エージェント | モデル | 役割 | 出力 |
|---|---|---|---|
| `planner` | opus | アイデア→製品仕様書。「何を作るか」のみを定義し、技術スタック等の実装詳細には踏み込まない | `docs/spec/<slug>-spec.md` |
| `generator` | sonnet | 指定された**1スプリント分のみ**を実装。合否は自分で判定せず、事実ベースの自己評価を残す | `docs/sprints/sprint-<N>-selfeval.md` |
| `evaluator` | opus | Playwright MCPでアプリを実際に操作して検証。**自分でコードは直さない**。基準を1つでも下回れば全体FAIL | `docs/sprints/sprint-<N>-evaluation.md` |

各エージェントは上記の役割以外に手を出さない設計。スプリント進行管理・PASS/FAILによるフロー分岐・gitコミットはオーケストレーター(呼び出し元Claude)側の責務。

## セキュリティ・品質ゲート

3エージェントの合否基準(致命的バグ0件/コンソールエラー0件/受け入れ基準充足率100%)は**機能面のみ**で、セキュリティやコード品質は含まれない。そのため`evaluator`がPASSした直後に、オーケストレーターが追加ゲートを実行してから初めてスプリントを確定させる。

1. `git add -A` でそのスプリントの変更をステージ(まだコミットしない)。
2. `git diff --cached --stat` で、`.gitignore`対象(`.env`・鍵ファイル・認証情報等)が誤ってステージされていないか確認し、あれば `git restore --staged` で除外する。
3. `/security-review`スキルをステージ済みの変更(`git diff --cached`)に対して実行する。
4. `/code-review`スキル(effort: `low`。Claude Codeに標準で組み込まれており追加コスト・外部依存なし)を同じ差分に対して実行し、evaluatorの機能テストにも`/security-review`のセキュリティ観点にも含まれない「正しさのバグ」「簡潔化・効率化の余地」を検出する。
5. 依存関係マニフェスト(`package.json`/`requirements.txt`等)を変更・追加した場合は、`npm audit`/`pip-audit`等の脆弱性スキャンも実行する。

- 上記いずれかでCritical/High相当の指摘、または`/code-review`が`CONFIRMED`と判定した正しさのバグ → そのスプリントをFAIL扱いにし、`git reset` でステージを解除した上で指摘内容を`generator`に渡して再実装(evaluatorのFAILと同じリトライ上限を共有)。
- 指摘なし、またはMedium/Low相当・`/code-review`の`PLAUSIBLE`判定や簡潔化提案等の軽微な指摘のみ → ステージ済みの変更を`git commit`してPASS確定、次のスプリントへ。

このレビューはgit差分ベースのため、プロジェクトはgitリポジトリで運用し、**スプリントが最終PASSした時点でオーケストレーターがコミットする**(スプリント境界=コミット境界)。generator/evaluatorはBashツールを持つがコミットは行わない。

## コード側の品質・安全性の最低ライン(generatorの実装原則)

- シークレット(APIキー・トークン・認証情報)はソースコードにハードコードせず、`.env`(gitignore済み)経由で読み込み、`.env.example`にキー名のみ残す。
- 外部API呼び出し・ファイルI/O・ユーザー入力など信頼境界のエラーハンドリングを実装する(起こり得ない事態への過剰な防御は書かない)。
- 依存関係は保守されている実績あるものを最小限だけ追加する。
- テストフレームワークが存在する場合、実装した機能に対する最小限のテストを追加し、`evaluator`はこれを含む既存テストスイートを実行してFAIL判定に含める。
- 起動方法・環境変数が変わったら`README.md`を最小限更新する。
- `planner`は、外部プラットフォームのAPI利用規約・レート制限・個人情報や認証情報の取り扱いなど無視できないリスクがあれば、仕様書の「リスク・留意事項」セクションに明記する。

## 運用ルール

- **リトライ上限**: 同一スプリントでFAILが続く場合、既定3回で「要人手介入」としてマークし、無限ループせず次のスプリントへ進む。理由なく上限を変更しない。
- **一時的な基盤障害とスプリントの合否は別カテゴリ**: 「temporarily unavailable」等、ツール呼び出し基盤側の一時的な瞬断による失敗は、コードの不具合ではないためリトライ上限を消費させず、FAILや要人手介入の根拠にもしない。同じ呼び出しを数回再試行すれば大抵解消する。詳細は`SKILL.md`の「一時的な基盤障害でスプリントを止めない・誤判定しない」参照。
- **スプリントは順序依存**: generator→evaluatorの順で呼び出し、並列実行しない。異なるスプリントを同時に進めない。
- **コンテキストの分断**: 各サブエージェント呼び出しは独立コンテキストで動く。呼び出すたびに`docs/sprints/sprint-<N>-brief.md`のパス（主な参照先）・spec.mdのパス・対象スプリント番号・直前のFAILフィードバックを明示的にプロンプトへ含める。
- **ブレインストーミング**: 新規アイデアが曖昧(対象プラットフォーム・ターゲットユーザー・スコープの境界等が読み取れない)な場合、`planner`呼び出し前にオーケストレーターがAskUserQuestionで2〜3問だけ確認する。詳細は明記済みの短いアイデアに毎回質問攻めにはしない。
- **多重起動の検知**: パイプライン開始前に`docs/pipeline-status.js`の`phase`/`lastUpdated`を確認し、別セッションが実行中の可能性がある場合(未完了フェーズかつ最終更新が直近)はユーザーに確認を取ってから開始する。同一フォルダへの複数セッション同時実行によるgit操作・状態ファイルの競合を防ぐため。詳細は`SKILL.md`の「前提: 多重起動の検知」参照。
- **Playwright MCP必須**: `evaluator` の実機検証には `.mcp.json` で定義した Playwright MCPサーバーが必要。ブラウザ本体未インストールが原因の失敗は`npx playwright install chromium`（無料・公式、管理者権限不要。保存先はユーザー単位の共有キャッシュ）を自動で試みてから再検証する。それでも失敗する場合のみ `Bash` でのAPI直叩き・静的確認に縮退し、evaluatorレポートの「検証モード」欄に明記した上でユーザーに報告する。
- **環境不足の自動補完**: ビルド・起動・テストが無料かつプロジェクトローカルなツール・パッケージの不足だけを理由に失敗する場合、`generator`/`evaluator`は「未検証」扱いにする前に自動でインストールしてから再試行する。管理者権限やOS全体への変更を要するインストールは対象外とし、その場合のみ手動インストール手順をレポートに残す。
- **事前確認（外部サービス依存・管理者権限インストール予想）**: (A)仕様書の「リスク・留意事項」に外部API連携・OAuth認証等ユーザー自身の操作が必要な項目があれば、または(B)対象プラットフォームが`デスクトップ`、もしくは機能一覧に動画/音声処理・重量級ローカルDB・OS常駐等ネイティブビルド/OSサービスを示唆する内容があり管理者権限インストールが予想される場合、スプリントループ開始前にオーケストレーターがAskUserQuestionで（両方該当すれば一度にまとめて）確認する。(A)は今すぐ用意するか揃うまでモック実装で進めるか、(B)は事前インストールするか管理者権限不要な代替手段で進めるか都度判断に委ねるかを選べる。(B)はあくまで予想であり外れても構わない（実際に必要になった時点の通常フォールバックに委ねる）。詳細は`SKILL.md`の該当手順参照。
- **サーバーライフサイクル**: `generator` は自己確認用に起動したサーバーを自己評価レポート作成前に必ず停止する。`evaluator` は自分で同じコマンドで起動し直し、ポートが使用中なら停止してから起動し直す。

## ディレクトリ規約

- `docs/spec/` — 製品仕様書(planner出力)
- `docs/spec/<slug>-wireframes.html` — 画面を持つプロダクトの場合にplannerが作成する主要画面のグレースケール・ワイヤーフレーム(配色・実装は含まない)。任意ファイル。
- `docs/sprints/` — スプリントごとの自己評価レポート・評価レポート(generator/evaluator出力)
- `docs/sprints/sprint-<N>-preview-*.png` — 画面を持つプロダクトでスプリントがPASSした際にevaluatorが保存するプレビュー画像(検証中に撮影済みのスクリーンショットの転用。追加処理・失敗時のブロックなし)。任意ファイル。
- `docs/dashboard.html` — 実行状況ダッシュボード(静的、書き換えない)。ブラウザで直接開くと、今どのエージェントが何をしているか・スプリント進捗・生成済みドキュメントへのリンクが30秒間隔で自動更新される（「今すぐ更新」リンクで即時更新も可能）。詳細は`SKILL.md`の「前提: ダッシュボード」参照。
- `docs/pipeline-status.js` — ダッシュボードのデータソース。オーケストレーターが状況変化のたびに上書きする実行時状態で、成果物ではないため`.gitignore`済み。
- `docs/sprints/sprint-<N>-brief.md` — 仕様書から対象スプリント分を原文のまま抜粋したファイル(要約ではない)。generator/evaluatorはこれを主な参照先とし、毎スプリント仕様書全体を読み直すトークンコストを避ける。
- `docs/project-memory.md` — プロジェクト全体の進捗・技術選定・既知の課題を集約した常設ダイジェスト。スプリントのPASS/要人手介入のたびに更新する。
- `docs/Home.md` — Obsidianで開いた際の入口(MOC)。上記ファイル群へのwikilinkを持つ。

## トークン節約の原則

いずれも**要約や省略による情報欠落ではなく**、原文の再配置・肥大化防止・冗長さの削減によるもので、成果物のクオリティを落とさない(むしろ曖昧さに起因する誤判定・手戻りを減らす分、品質にも寄与する)。

- **原文抜粋の再利用**: `sprint-<N>-brief.md`(仕様書からの抜粋)を主な参照先にし、仕様書全体の読み直しを避ける。
- **不合格理由は要点だけを転記**: リトライ時、オーケストレーターはevaluatorレポートの「発見したバグ・問題点」や`/security-review`のCritical/High指摘そのものだけを次のgenerator呼び出しに転記する。PASSした基準の一覧など修正に不要な部分は転記・再読させない。
- **`docs/spec/<slug>-architecture.md`は追記でなく上書き**: 同じ領域の技術選定を再検討したら古い記載を置き換える。決定の変遷ではなく「現在の決定」だけを保持し、プロジェクトの長さに比例した肥大化を防ぐ。
- **レポートは簡潔な箇条書き**: generator/evaluatorの各種レポートは長いプローズを避ける。再試行時に他のエージェントが読む/転記することを前提に書く。
- **受け入れ基準は具体的・検証可能に**: 曖昧な基準は誤判定や無駄な再実装サイクルに直結するため、plannerの時点で手順として読める粒度まで具体化する。
