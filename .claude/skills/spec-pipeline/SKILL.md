このスキルは `planner` / `generator` / `evaluator` の3つのサブエージェント（`.claude/agents/`配下）を、Agentツールで順番に呼び出してオーケストレーションする。あなた（呼び出し元のClaude）がループの状態管理・git操作・`/security-review`と`/code-review`スキルの呼び出しを担う。各サブエージェントは自分の役割以外に踏み込まない設計なので、スプリント間の進行管理・合否によるフロー分岐・コミットはこのスキルの責務。

## 最重要原則: 記録・可視化のための処理は本来の進捗を絶対に止めない

このスキルには、本体の進行（`planner`→`generator`→`evaluator`→レビュー→コミットのスプリントループ）と、それを補助する副次的な記録・可視化の処理（`docs/pipeline-status.js`・`docs/project-memory.md`・`docs/Home.md`の更新、多重起動チェックなど）の2種類の作業が登場する。**後者は前者より常に優先度が低く、後者の不具合が前者を遅延・停止させることは絶対に許容しない。**

- 記録・可視化の更新でエラーが起きた場合（Editツールの文字列不一致、ファイル読み取り失敗など）、原因調査や修正に時間をかけない。**1回だけ修正を試み、それでも直らなければその回の更新は諦めて（記録が1回分欠けたままで）本体のスプリントループを続行する。** 記録の欠落は実害がないが、本体の進行が止まる方が実害が大きい。
- 記録・可視化の都合でユーザーに確認を求めたり処理を一時停止したりしない。ユーザー確認が必要なのは「前提: 多重起動の検知」で明示した1箇所（パイプライン開始前）だけであり、それ以外の記録処理では発生しない。
- `docs/project-memory.md`・`docs/Home.md`の更新は、Editツールの完全一致要件によるエラーを避けるため、**部分編集(Edit)より全文の書き直し(Write)を基本とする**（既存の構成・説明文は省略せずそのまま含めて書き直す。詳細は該当セクション参照）。

## 最重要原則: 一時的な基盤障害でスプリントを止めない・誤判定しない

Agentツール・Skillツール・Bash/Edit/Write等の呼び出しが、コードや設定の不備ではなく「ツール呼び出し基盤側の一時的な瞬断」（例:「temporarily unavailable」「rate limit」「timeout」のようなエラーで、モデルやツール実行基盤が一時的に応答不能になる事象）で失敗することがある。**これはスプリントの実装・検証結果とは無関係な偶発的事象であり、上の「記録・可視化」の不具合とも異なる第三のカテゴリとして扱う。**

- この種の失敗は**同じ呼び出しをそのまま（内容を変えずに）数回再試行すれば大抵は解消する**（このテンプレート自体の開発中にも複数回発生し、そのつど再試行だけで解消した実績がある）。原因調査や設定変更、回避策の模索に時間をかけない。
- **スプリントのリトライ上限（3回）を絶対に消費させない。** evaluatorのFAILや`/security-review`・`/code-review`の指摘とは性質が異なり、ジェネレーターに実装をやり直させるべき事象ではないため。同様に、これを理由に「要人手介入」としてマークしたり、スプリントをFAIL扱いにしたりしない。
- `generator`・`evaluator`・`/security-review`・`/code-review`の呼び出しなど**本体の進行そのもの**がこの種のエラーに遭遇した場合は、目安5回程度は無言で再試行する。それでも解消しなければユーザーに「ツール呼び出し基盤が一時的に不安定なようです。しばらく時間を置いて自動的に再試行します」と一言状況を共有した上で、なお再試行を続ける（ここで諦めて停止しない）。目安20回、あるいは体感で数分以上続けても解消しない場合のみ、無限に再試行し続けるのではなく「基盤障害が長引いている可能性があるため、しばらく時間を置いてからの再開を推奨します」とユーザーに伝えて一旦待機する。ユーザーから明示的な停止指示があれば直ちに従う。
- `docs/pipeline-status.js`等の記録・可視化処理がこの種のエラーに遭遇した場合も同様に数回は再試行してよいが、それでも解消しなければ「記録・可視化のための処理は本来の進捗を絶対に止めない」の原則に従い、記録を諦めて本体の進行を優先する。

## 前提: gitリポジトリ

このパイプラインはスプリント境界をgitコミットで区切り、`/security-review`スキル（git差分ベース）でセキュリティレビューを行う。プロジェクトがgitリポジトリでない場合は、パイプライン開始前にユーザーに `git init` の許可を取る（git configのuser.name/user.emailが未設定なら、ユーザー自身に設定してもらう。Claude側からgit configは変更しない）。

## 前提: 多重起動の検知（競合防止）

同じプロジェクトフォルダに対して2つ以上のセッションが同時に`/spec-pipeline`を実行すると、git操作（`add`/`commit`/`stash`）や`docs/pipeline-status.js`・`docs/project-memory.md`・`docs/Home.md`への書き込みが競合し、コミット破損やステータスの上書き事故につながる。**このスキルを開始する前に必ず次を確認する**（新規アイデア・既存specからの再開のいずれでも）:

- `docs/pipeline-status.js`が存在すれば`phase`と`lastUpdated`を読む。`phase`が`idle`・`completed`のいずれでもなく、かつ`lastUpdated`が現在時刻からおおむね10分以内であれば、**同じフォルダで別のセッションが今まさに実行中である可能性が高い**。この場合、処理を開始せずユーザーに「`docs/pipeline-status.js`の状態から、別セッションがこのプロジェクトを実行中の可能性があります（最終更新: 〜分前、フェーズ: 〜）。本当にこのセッションでも開始しますか？」と確認を取り、明示的な許可を得てから続行する。
- 上記チェックは`docs/pipeline-status.js`が存在しない、または`phase: "idle"`/`"completed"`であれば即座にPASSとし、通常通り開始する（毎回ユーザーに確認を強いるものではない）。
- **このチェック自体が読み取り・解釈に失敗した場合（ファイルが壊れている等）は、安全側ではなく「進行を止めない側」に倒す。** 判断材料が得られないだけであり、通常通り開始してよい（原因調査に時間をかけない）。

## 前提: ダッシュボード（`docs/dashboard.html`）

`docs/dashboard.html` と `docs/pipeline-status.js` はテンプレートに同梱済みの実行状況ダッシュボードで、ユーザーがブラウザで直接開いて（`file://`でよい、サーバー不要）今どのエージェントが何をしているか・スプリント進捗・生成済みドキュメントへのリンクを確認できる。30秒ごとに自動リロードされ、「今すぐ更新」リンクで即座にも更新できる。

- **`docs/dashboard.html` は静的ファイルであり、書き換えない。** 表示ロジック・デザインはこのファイルに固定済み。
- **`docs/pipeline-status.js` だけを、状況が変わるたびに`Write`ツールで丸ごと上書きする。** これがダッシュボードの唯一のデータソース（スキーマはファイル冒頭のコメントを参照）。小さなJSファイルの全文上書きなので書き換えコストは低く、失敗もしにくい。万一この更新に失敗しても構わず先へ進む（「最重要原則」参照）。
- 更新すべきタイミング（詳細は各手順内に記載）: パイプライン開始時／planner呼び出し前後／各スプリントでgenerator・evaluator・security-review・code-reviewを呼ぶ前後／PASS・FAIL・要人手介入・コミット確定時／全スプリント完了時。`currentActivity`と`recentEvents`は必ず最新化し、`lastUpdated`を現在時刻(ISO8601)に更新する。generatorが`docs/spec/<slug>-architecture.md`を新規作成した回があれば、そのタイミングで`architecturePath`も記録する。
- `docs/pipeline-status.js` は実行時状態であり成果物ではないため`.gitignore`済み。スプリントコミットに混ざらない（オーケストレーターが意識して除外する必要はない）。
- ユーザーには、パイプライン開始直後に一度だけ「進捗は `docs/dashboard.html` をブラウザで開くとリアルタイムに確認できます」とファイルパスを明示して案内する（毎スプリントで繰り返し案内しない）。**オーケストレーターは`docs/dashboard.html`を自らブラウザで開くことは絶対にしない**（`Bash`でのブラウザ起動コマンド実行や、Playwright MCP等でのナビゲーションを含む）。案内はテキストでパスを伝えるだけに留め、実際に開くかどうかはユーザー自身が判断する。開いたあとの自動更新（30秒間隔・「今すぐ更新」リンク）は`docs/dashboard.html`自身のブラウザ内JavaScriptが行うものであり、オーケストレーター側の追加操作は不要。

## 前提: ドキュメント構成とトークン節約（`sprint-<N>-brief.md` / `project-memory.md` / `Home.md`）

サブエージェント呼び出しは独立コンテキストで動くため、素朴に実装すると毎スプリント`generator`・`evaluator`が仕様書全体を読み直すことになりトークンを消費する。これを避けるため次の3ファイルを維持する。**いずれも要約ではなく原文の再配置・抜粋であり、情報の欠落や成果物の品質低下は発生しない**（受け入れ基準・評価基準は一言一句そのまま転記する）。

- **`docs/sprints/sprint-<N>-brief.md`**: 仕様書の対象スプリント部分（含まれる機能の説明・受け入れ基準・評価基準・対象プラットフォームを原文のまま）を抜粋したファイル。plannerの結果を受け取った直後に全スプリント分をまとめて1回だけ生成する（オーケストレーターが仕様書を1回読むだけで済み、以後generator/evaluatorは仕様書全体ではなくこの小さなファイルを読めばよくなる。詳細はagent定義側にも記載済み）。
- **`docs/project-memory.md`**: プロジェクト全体の進捗・技術選定・既知の課題を1ファイルに集約した、常に最新化されるダイジェスト。スプリントのPASS/要人手介入確定のたびに更新する。人間がObsidian/NotebookLM等でプロジェクト全体を素早く把握するための一次情報源でもある。
- **`docs/Home.md`**: Obsidianで開いたときの入口となるMOC(Map of Content)。spec.md・architecture.md・project-memory.md・各スプリントのレポートへのリンクを持つ。テンプレートに同梱済みの初期ファイルを編集し、スプリントが進むたびにリンクを追記する。

これらはユーザーがObsidianでプロジェクトフォルダ（または`docs/`）をVaultとして開けば、wikilinkとbacklinkで一覧・検索できる。Obsidianを使わなくても普通のMarkdownとして問題なく機能するため、常に生成してよい。

**`docs/project-memory.md`と`docs/Home.md`はテンプレートに初期状態（プレースホルダーと説明文付き）で同梱済みであり、新規に作成するファイルではない。** 更新するたびに、既存の構成・見出し・説明文（NotebookLMの補足など）は省略せずそのまま含め、プレースホルダー部分（「（未生成。...）」等）だけを実データに置き換えた全文を**`Write`ツールで書き直すことを基本とする**（Editツールの完全一致要件による書き換え失敗・手戻りを避けるため。ファイル全体を毎回自分の手元で組み立て直すだけなので、Editより確実で速い）。

**NotebookLMには公開APIが無く、個人向けの連携手段は非公式ツール（Googleの内部APIをリバースエンジニアリングし、ユーザーのGoogleセッションで認証するもの）に限られる。このパイプラインが`docs/project-memory.md`等をNotebookLMへ自動送信することは一切行わない**（プロジェクトのドキュメントを外部の信頼できない経路へ自動的に送出する仕組みは組み込まない。ユーザーの同意があっても行わない）。NotebookLMを使いたい場合は、ユーザーが`docs/project-memory.md`等を都度手動でnotebooklm.google.comにアップロードする（詳細はREADME.md参照）。

## 全体フロー

```
[新規アイデアの場合]
  0. アイデアが曖昧なら、planner呼び出し前にAskUserQuestionで2〜3問だけ確認する（ブレインストーミング）
  1. planner を1回呼ぶ → docs/spec/<slug>-spec.md（対象プラットフォームに画面があればwireframes.htmlも）が生成される

[スプリントループ: spec.md の各スプリントについて]
  2. generator を呼ぶ（対象スプリント番号 + 前回のFAILフィードバックがあれば渡す）
  3. evaluator を呼ぶ（対象スプリント番号） → FAILなら2に戻る（リトライ）
  4. evaluatorがPASSなら、そのスプリントの変更（pending changes）に対して `/security-review` と `/code-review` を実行
     - Critical/High相当の指摘、または`/code-review`のCONFIRMEDな正しさのバグあり → FAIL扱いとし、指摘内容を持って2に戻る（同じリトライ上限を消費）
     - 指摘なし、またはMedium/Low相当・簡潔化提案等の軽微な指摘のみ → そのスプリントを最終PASSとし、変更をコミットして次のスプリントへ
  5. 全スプリント終了後、サマリーをユーザーに報告する
```

## 実行手順

1. **入力の判定**: まず**多重起動の検知**（前提セクション参照）を行い、問題なければ続行する。引数が短い自然文（1〜4行）なら新規アイデアとして扱う。
   - **ブレインストーミング（新規アイデアの場合のみ。既存specへのパスが渡された場合はアイデア文自体が無いためこの手順は適用外）**: アイデアの中で、対象プラットフォーム・主要ターゲットユーザー・スコープの境界（何を含め何を含めないか）・（該当しそうなら）収益化方法のいずれかが読み取れない、または解釈が大きく割れうると判断した場合、`planner`を呼ぶ前に**AskUserQuestion**でユーザーに2〜3問だけ確認する。既にアイデア文にこれらが明記されている、または短くても曖昧さがないと判断できる場合はこの手順を省略し、質問攻めにしない。ここで曖昧さを解消しておくことで、`planner`が誤った前提で仕様を膨らませて後工程（スプリント丸ごとの手戻り）を招くリスクを減らす。得られた回答をアイデアに統合してから次に進む。
   - `docs/pipeline-status.js` を初期化（`phase: "planning"`, `currentActivity: {agent: "planner", label: "製品仕様書を作成中"}`）してから、`planner` エージェントをAgentツールで呼び出す。既に `docs/spec/*.md` へのパスが渡された場合はplannerをスキップし、そのspecを使う（この場合も`docs/pipeline-status.js`は同様に初期化する）。`docs/project-memory.md` が存在する場合は、途中経過の把握のため必ず読んでから作業を開始する（完了済みスプリント・直前のFAIL理由・未解決事項を踏まえて再開する）。
2. **plannerの結果を受け取る**: 生成されたspec.mdのパスと、スプリント数、および「対象プラットフォーム」（web/モバイル(Expo)/デスクトップ）を把握する。TodoWriteでスプリント一覧をタスク化し、対象プラットフォームを併記して進捗を可視化する（Sprint 1〜N、各ステータスは pending/in_progress/pass/fail-escalated）。`docs/pipeline-status.js` も同じ内容で更新する（`projectName`・`specPath`・`platform`・`phase: "sprint_loop"`・`sprints`配列を仕様書のスプリント計画から生成、各`status: "pending"`・`attempts: 0`・`maxAttempts`は適用中のリトライ上限（既定3、ユーザー指定があればその値）・`commit: null`・`previewPaths: []`で初期化）。`recentEvents`に「製品仕様書を作成しました」を追記。ユーザーには最初の1回だけ「進捗は `docs/dashboard.html` で確認できます」と案内する。
   - `docs/sprints/sprint-<N>-brief.md` がまだ存在しないスプリントについて、仕様書を読んで全スプリント分をこの時点でまとめて生成する（既存プロジェクトを既存specから再開する場合も同様に、無ければここで生成する）。内容は該当スプリントの「含まれる機能」の説明・受け入れ基準・「評価基準」・「対象プラットフォーム」を仕様書から**原文のまま**転記する（要約しない）。
   - テンプレート同梱済みの`docs/project-memory.md`（プレースホルダー部分のみ）を、概要・対象プラットフォーム・スプリント一覧で更新する。同じくテンプレート同梱済みの`docs/Home.md`の「## 仕様」プレースホルダーをspec.mdへのwikilinkで、その他必要な箇所をproject-memory.mdへのリンクで更新する（「## スプリント」欄はまだ空のままでよい）。`docs/spec/<slug>-wireframes.html`が存在すれば（新規生成・既存プロジェクト再開のいずれでも）、そのリンクを`docs/Home.md`の「## 仕様」欄と`docs/pipeline-status.js`（`wireframesPath`）の両方に追加する。
   - `README.md`がテンプレート由来の汎用的な説明のまま（例: 見出しや本文に「spec-pipeline テンプレート」等、このプロダクト固有でない文言が残っている）であれば、仕様書の「概要」セクションを基にオーケストレーターがプロダクト名・概要で該当箇所を書き換える。起動方法・環境変数の欄は空のまま残し、generatorが最初のスプリントで反映できるようにする（generator/planner自身の役割は変えない）。既にプロダクト固有の内容になっていれば何もしない。
   - **事前確認（外部サービス依存・管理者権限インストール予想）**: 以下の2種類の確認事項をチェックし、いずれか一方でも該当すればスプリントループを開始する前にAskUserQuestion等で**一度にまとめて**確認する（該当が1種類だけならその1問だけ、両方あれば2問まとめて。ユーザーへの割り込みを1回に集約するため）。いずれも該当しなければこの手順自体を省略する（毎回質問攻めにしない）。
     - **(A) 外部サービス依存**: 仕様書の「リスク・留意事項」セクションに、外部プラットフォームAPI連携・OAuth認証・決済等、**ユーザー自身の操作（ログイン・APIキー発行・アカウント登録等）が必要な項目**が含まれる場合、「この仕様には次の外部サービス連携が含まれ、稼働にはあなたの操作が必要です: <一覧>。(a) 今このタイミングで認証情報等を用意する、(b) 用意ができるまでは該当箇所をモック/スタブ実装で進め、揃い次第本接続に切り替える、どちらにしますか？」と確認する。(a)が選ばれ実際に情報を受け取った場合は`.env`への設定方法を案内する（値そのものをチャットに貼らせた場合は`.env`に書いたことを確認し、チャット上に平文で残さないよう促す）。(b)が選ばれた、またはユーザーが未対応の場合は、仕様書の「スプリント計画」を見てその外部サービスに関わる機能（F#）を含む**全てのスプリント番号**（1つとは限らない）と「モック実装で進める」方針を`docs/project-memory.md`の「既知の課題・要人手介入」に記録し、該当する各スプリントのgenerator呼び出し時にその指示をプロンプトに含める。
     - **(B) 管理者権限インストールの予想**: `generator`/`evaluator`の「環境不足の自動補完」は無料・プロジェクトローカルな範囲に限られ、管理者権限を要するインストールは自動実行されない（各エージェント定義参照）。仕様書の「対象プラットフォーム」が`デスクトップ`である場合（ネイティブモジュールのビルドツール、例えばWindowsのMicrosoft C++ Build Tools・MacのXcode Command Line Tools等が必要になりやすい）、または「機能一覧」に動画/音声のローカル処理・重量級ローカルDB（PostgreSQL/MySQL等の常駐サービス）・OSトレイ常駐・Bluetooth/USB等ハードウェア直接アクセスのような、通常ネイティブビルドツールやOSサービスのインストールを伴う内容が含まれる場合、その工程は高い確度で管理者権限インストールを要すると**予想**されるため確認する:「実装の過程で、次の工程は管理者権限でのインストールが必要になる可能性があります: <予想項目とその理由>。(a) 該当ツールを今のうちにご自身でインストールしておく、(b) 管理者権限が不要な代替手段（クラウドサービス・Docker・純npm実装等）で進める方針にする、(c) 実際にその工程に到達した時点で都度判断する、のどれにしますか？」。(a)なら「インストール済み」の申告を待ってから該当スプリントに進めてよい。(b)なら該当する**全てのスプリント番号**（1つとは限らない）と「管理者権限インストール回避・代替手段で進める」方針を`docs/project-memory.md`の「既知の課題・要人手介入」に記録し、該当する各スプリントのgenerator呼び出し時にその指示をプロンプトに含める。(c)または未回答なら特別な記録はせず通常通り進め、実際にその場面に到達したときは既存の「環境不足の自動補完」のフォールバック（手動インストール手順をレポートに明記）に従う。この確認はあくまで**予想**であり、外れる（実際には管理者権限が不要だった、または予想しなかった別の箇所で必要になった）ことがあっても構わない。予想が外れたこと自体を問題視せず、実際に管理者権限インストールが必要になった時点の通常フォールバックに委ねる。
3. **スプリントループを開始する**（開始番号の指定があればそこから、なければSprint 1から）。各スプリントについて:
   - `docs/pipeline-status.js` の該当スプリントの`attempts`を1つインクリメントした上で（初回呼び出しなら0→1）、`status: "in_progress"`、`currentActivity`を`{agent: "generator", sprintNumber: N, label: "Sprint N の実装中（試行<インクリメント後のattempts>/<maxAttempts>）", startedAt: 現在時刻}`に更新してから、`generator` をAgentツールで呼び出す。プロンプトには次を含める: **`docs/sprints/sprint-<N>-brief.md`のパス（主な参照先）**、spec.mdのパス、対象スプリント番号、（再試行の場合）直前の不合格理由。**不合格理由はレポートのパスだけを渡すのではなく、orchestrator自身がevaluatorレポートの「発見したバグ・問題点」節（該当すれば「軽微な改善点」も）、または`/security-review`・`/code-review`のCritical/High相当の指摘そのものを読み取り、その部分だけをプロンプト本文に転記する**（PASSした基準の一覧・検証モード欄など修正に不要な部分は転記しない。generatorに全文を読み直させないことで、トークンを使わずに同じ情報量を伝える）。
   - generatorから自己評価レポートのパスを受け取ったら、`docs/pipeline-status.js` の該当スプリントに`selfEvalPath`を記録し`currentActivity`を`{agent: "evaluator", label: "Sprint N を実機検証中"}`に更新する。**この時点で`docs/spec/<slug>-architecture.md`が今回のgenerator呼び出しで新規に作成されていれば**（呼び出し前には無く、今は存在する場合。generatorの自己評価レポートの「技術選定」節でも判別できる）、`docs/pipeline-status.js`の`architecturePath`と`docs/Home.md`のリンク一覧にこのファイルを追加する。その上で続けて `evaluator` をAgentツールで呼び出す。プロンプトには: `docs/sprints/sprint-<N>-brief.md`のパス（主な参照先）、spec.mdのパス、対象スプリント番号、generatorの自己評価レポートのパスを含める。**このスプリントが「事前確認」でモック/スタブ実装の対象になっていた場合は、その旨と対象機能も併せて伝える**（例:「F3のGoogleログインは認証情報待ちのためモック実装です。モックとして一貫して動作すれば十分とし、本物の外部サービス連携としては判定しないでください」）。これを伝え忘れると、evaluatorが正規の受け入れ基準（本物の外部サービス連携を前提に書かれている）でモック実装を判定し、承知の上の未実装を理由に不当にFAILさせてしまう。
   - evaluatorがFAILなら、`docs/pipeline-status.js` の該当スプリントを`status: "fail_retry"`・`lastIssue`に理由を一言で記録し、`recentEvents`に追記した上でリトライ制御へ（4.を経ずに2に戻る）。
   - evaluatorがPASSなら、該当スプリントに`evaluationPath`を記録し、evaluatorの報告に`sprint-<N>-preview-*.png`（プレビュー画像）があれば`previewPaths`にも記録する。`currentActivity`を`{agent: "orchestrator", label: "Sprint N のセキュリティレビュー準備中"}`に更新した上で、`git add -A` でそのスプリントの変更をステージする（まだコミットしない）。
   - `git diff --cached --stat` でステージされたファイル一覧を確認し、`.gitignore`で除外されるべきファイル（`.env`、鍵ファイル、認証情報など）が紛れ込んでいないか確認する。紛れ込んでいれば `git restore --staged <file>` で除外してから先へ進む。
   - `currentActivity`を`{agent: "security-review", label: "Sprint N のセキュリティレビュー中"}`に更新し、`/security-review` スキルをSkillツールで呼び出し、ステージされたそのスプリントの変更（`git diff --cached`）を対象にレビューさせる。
   - 続けて`currentActivity`を`{agent: "code-review", label: "Sprint N のコードレビュー中"}`に更新し、`/code-review` スキルを（effort: `low`、無料かつ既にClaude Codeに組み込まれているスキルで追加の外部依存なし）ステージされた同じ差分に対して実行し、機能面の評価軸（evaluator）にもセキュリティ観点（`/security-review`）にも含まれない「正しさのバグ」「簡潔化・効率化の余地」を検出する。
   - 依存関係マニフェスト（`package.json`/`requirements.txt`等）が今スプリントで変更・追加されていれば、`Bash`で該当エコシステムの脆弱性スキャン（`npm audit`/`pip-audit`等）を実行し、結果を合わせて判定材料にする。
   - Critical/High相当の指摘（`/security-review`の指摘、脆弱性スキャン、除外しきれない機密情報、`/code-review`が`CONFIRMED`とした正しさのバグのいずれか）があれば、そのスプリントをFAIL扱いにする。`docs/pipeline-status.js`を`status: "fail_retry"`・`lastIssue`更新の上、`git reset` でステージを解除（作業ツリーの変更は保持）した上で、該当する指摘そのもの（レビュー出力全文ではない）を次のgenerator呼び出しのプロンプトに含めてリトライ制御へ。`/code-review`の`PLAUSIBLE`判定（確証はないが疑わしい）は即FAILにはせず、次点の「軽微な改善点」としてPASS後に記録する。
   - 指摘なし、またはMedium/Low相当・簡潔化提案等の軽微な指摘のみなら、コミット前に`docs/project-memory.md`（該当スプリントをPASS済みとして記録・技術選定や既知の課題を更新）と`docs/Home.md`（このスプリントのレポート・プレビュー画像（あれば）へのリンクを追記）を更新する。その上で`git add -A`し直し（project-memory.md/Home.mdの更新分を含めるため）、`git commit -m "Sprint <N>: <スプリント目標>"` でステージ済みの変更をコミットし、そのスプリントを最終PASSとしてTodoWriteを更新、次のスプリントに進む。`docs/pipeline-status.js`の該当スプリントを`status: "pass"`・`commit`にコミットハッシュを記録し、`summary.passed`をインクリメント、`recentEvents`に「Sprint N PASS（コミット済み）」を追記する。
4. **リトライ制御**: 同一スプリントで（evaluatorのFAILであれ`/security-review`・`/code-review`のFAILであれ）不合格が続く場合、合計の上限（デフォルト3回、ユーザー指定があればそれに従う）に達したらそのスプリントを「要人手介入」としてマークする。作業ツリーに残ったそのスプリントの変更は `git stash push -u -m "Sprint <N> 要人手介入: <直近の不合格理由を一言>"` で退避してから次のスプリントに進む（コミットはしない）。これを怠ると、次スプリント以降の `git add -A` に未解決の変更が混入し、スプリント境界=コミット境界の原則が崩れて未解決コードが後続スプリントのコミットに紛れ込む。理由もなく上限を増減させない。`docs/pipeline-status.js`の該当スプリントを`status: "human_intervention"`に更新し`summary.humanIntervention`をインクリメント、`recentEvents`に理由とともに追記する。`docs/project-memory.md`の「既知の課題・要人手介入」にも同じ理由を記録する（stashは既にコミット前なので、この更新は次スプリントの`git add -A`で自然にコミットされる）。
5. **進捗報告**: 各スプリントのPASS/FAIL確定時に、簡潔な一文でユーザーに進捗を共有する（詳細なレポート全文の垂れ流しはしない。パスを示せば十分）。プレビュー画像が保存されたスプリントでは、そのパスも一言添える。
6. **完了時のサマリー**: 全スプリント終了後、`docs/pipeline-status.js`を`phase: "completed"`・`currentActivity: {agent: "idle", label: "全スプリント完了"}`に更新した上で、以下を報告する:
   - 総スプリント数、PASS数、要人手介入数
   - 要人手介入となったスプリントとその理由（evaluatorレポート/security-review指摘へのパスまたは要約）。作業ツリーの変更を`git stash`で退避した場合は、`git stash list`で参照できる旨とスタッシュメッセージ（スプリント番号）を明記する
   - evaluatorレポートの「検証モード」がBash縮退だったスプリント（あれば）とその番号。実機検証されていないため、後日Playwright MCP設定後の再検証を推奨する旨を添える
   - 各スプリントのevaluatorレポートに「未検証項目（実機確認が必要）」の記載があれば（検証モードがBash縮退か否かに関わらず）、スプリント番号とあわせて一覧化し、実機での確認をユーザーに促す
   - `/security-review`でMedium/Low指摘、または`/code-review`で簡潔化・効率化提案等の軽微な指摘があったが非ブロッカーとしてPASSさせたスプリント（あれば）とその概要
   - 次にユーザーが確認・対応すべきこと

## ネイティブアプリのストア公開フェーズ（該当する場合のみ）

spec.mdの対象プラットフォームが`モバイル(Expo)`で、かつユーザーがApp Storeへの公開を希望する場合、全スプリントの最終PASS後に本フェーズを行う。generator/evaluatorのスコープ外であり、オーケストレーター（このスキルの呼び出し元）が直接進める。

1. **ユーザーに依頼する一度きりの事前準備（代行不可）**:
   - Apple Developer Programへの登録（年間$99、Apple ID・支払い・本人確認はユーザー自身で行う必要がある）。
   - App Store Connect API Key（.p8ファイル、Key ID、Issuer ID）の発行。App Store Connectのポータルで一度だけユーザー自身がログインして生成し、ファイルパスを教えてもらう。鍵の中身はチャットに貼らせず、`.env`等gitignore対象のローカルファイルとして扱う。
2. **上記が揃い次第、オーケストレーターが実行する**:
   - EAS CLI導入、`eas.json`等のビルド設定作成、`eas build --platform ios`でのクラウドビルド実行（Mac不要）。
   - App Store Connect API経由（EAS Submit等）で、Bundle ID登録・アプリレコード作成（アプリ名/SKU/主言語）・ストア掲載文面（説明文・キーワード・サポートURL・プライバシーポリシー文面）・年齢レーティング申告・アプリプライバシー申告の登録。
   - Web版レンダリングを基にしたApple指定サイズのスクリーンショット生成。
   - ビルドのApp Store Connectへのアップロード。
3. **審査提出（`eas submit`等）の直前に、登録した掲載内容・スクリーンショットのサマリーをユーザーに見せて一度確認を取る。** 提出は審査枠を消費し取り消しにくい操作のため、無断で実行しない。
4. Appleからの審査結果（却下・要修正等）が来た場合は内容をそのままユーザーに報告し、コンテンツ/ポリシー判断が必要な却下理由であればユーザーの判断を仰いでから対応する。

## 注意事項

- 各エージェント呼び出しは独立したコンテキストで動くため、Agentツールのプロンプトには必要な情報（spec.mdのパス、スプリント番号、直前のフィードバック）を毎回明示的に含めること。前の呼び出しの記憶には頼れない。
- generator/evaluatorの呼び出しは順序に依存する（evaluatorはgeneratorの成果物を前提とする）ため、並列実行しない。異なるスプリントを並列で進めることもしない（スプリント間に依存関係があるため）。
- Playwright MCPが未設定の環境では evaluator が実機検証を縮退させる場合がある。その場合はユーザーにMCPサーバーの設定を促す。
- **git操作（add/commit）はオーケストレーター（このスキル）のみが行う。** generator/evaluatorにBashツールがあってもコミットさせない。スプリント境界の一貫性が崩れ、`/security-review`の差分範囲がずれるため。
- ユーザーから明示的な停止指示があれば、現在進行中のスプリントの区切りで停止する。
- `docs/pipeline-status.js`の更新（Writeツールでの上書き）は軽量なファイル書き込みのみで、サブエージェント呼び出しを伴わない。スプリント本体の処理速度に影響しないため、更新を惜しんで省略しない。
- `docs/sprints/sprint-<N>-brief.md`・`docs/project-memory.md`・`docs/Home.md`・`docs/sprints/sprint-<N>-preview-*.png`は`docs/pipeline-status.js`と違い実行時の使い捨てファイルではなく、正式なプロジェクト成果物として通常通りコミット対象に含める（`.gitignore`されているのは`docs/pipeline-status.js`と`.playwright-mcp/`スクラッチディレクトリだけ）。
- NotebookLMには公開APIが無いため、オーケストレーターがそこへ自動でデータを送信することはない。ユーザーが手動で行う運用としてREADME.mdに案内するのみ。
- **環境不足への対応はgenerator/evaluator自身が行う**（無料・プロジェクトローカルな範囲での自動インストール。詳細は各エージェント定義参照）。オーケストレーターはこれを前提とし、「ツールが無いので手動インストールしてください」という報告をそのままユーザーに転送する前に、自己評価/評価レポートに実際に自動インストールを試みた記載があるか確認する。
- プレビュー画像（`docs/sprints/sprint-<N>-preview-*.png`）はevaluatorが直前まで検証していた画面状態のまま追加撮影・保存するだけであり、新しいブラウザナビゲーションや追加の検証操作を発生させない。生成の有無に関わらずスプリントの合否・進行には影響しない。
